你现在是一名资深前端产品设计师 + UI 工程师，请为一个“AI
  多模型网页聊天系统”生成前端界面原型/静态页面，重点先完成界面层，并且页面结构要便于后续接入真实后端接口。

  这个项目的前端聊天消息发送，统一采用流式返回（SSE），不需要实现全量返回模式，请围绕这一点进行界面设计。

  ---
  一、产品背景

  这是一个类似 ChatGPT / Claude / Gemini 风格的网页聊天产品，后端已经完成，是一个基于自研 ChatSDK 的 HTTP
  服务。当前目标是先生成前端界面，整体要求：

  1. 简约风格
  2. 布局参考主流大模型网页聊天界面
  3. 页面观感干净、留白充足、层级清晰
  4. 支持桌面端优先，同时兼顾响应式
  5. 前端先以“可直接连调的聊天界面骨架”为目标
  6. 消息发送统一为流式返回，不设计普通一次性返回模式

  ---
  二、页面整体布局要求

  请生成一个单页面聊天应用，采用主流 AI 聊天产品的布局方式：

  1）左侧边栏：会话区

  用于展示会话列表和管理会话，包含以下元素：

  - 顶部应用标题，例如：AI Chat
  - 一个醒目的主按钮：“新建会话”
  - 会话列表区域
    - 每个会话项显示：
        - 会话标题（可优先显示第一条用户消息摘要，没有则显示“新对话”）
      - 当前模型名称
      - 更新时间
      - 消息数量
    - 当前选中的会话高亮
    - 每个会话项右侧或悬浮时出现删除按钮：“删除”
  - 底部可保留轻量状态区，展示：
    - 当前已选模型
    - 连接状态（如“后端已连接 / 未连接”）

  2）顶部工具栏：模型区

  位于主聊天区域顶部，包含：

  - 当前会话标题
  - 模型选择下拉框：“选择模型”
  - 次级按钮：“刷新模型列表”
  - 一个轻量标签或状态提示：“当前模式：流式回复”
  - 不需要提供“普通回复 / 流式回复”切换，因为本项目统一使用流式回复

  3）中间主区域：聊天消息区

  仿照主流大模型聊天页面，要求：

  - 内容区居中，宽度适中
  - 消息按时间顺序垂直排列
  - 区分用户消息与助手消息：
    - 用户消息使用不同背景或靠右布局
    - 助手消息靠左
  - 每条消息显示：
    - 角色
    - 内容
    - 时间戳
  - 助手回复必须支持流式生成中的视觉状态
  - 空状态时展示欢迎区：
    - 欢迎文案
    - 提示“请选择模型并开始对话”
    - 3 个示例问题卡片按钮（仅作 UI 占位）

  4）底部输入区：消息发送区

  固定在底部，仿照 ChatGPT 风格，包含：

  - 一个多行输入框
  - 主按钮：“发送”
  - 次按钮：“停止”
  - 一个小提示文案：
    - 当前模式：流式回复
  - 输入框附近增加轻量说明：
    - 内容由 AI 生成，请注意甄别

  ---
  三、必须包含的按钮与控件

  请在界面中明确生成以下按钮/控件，并保证命名清晰：

  1. 新建会话
  2. 删除
  3. 刷新模型列表
  4. 发送
  5. 停止
  6. 选择模型（下拉框）

  注意：

  - 不要生成“普通回复”按钮
  - 不要生成“流式回复”切换按钮
  - 页面中只保留“当前模式：流式回复”的只读状态展示即可

  ---
  四、后端接口约束（前端设计必须围绕这些接口）

  后端当前已实现的接口如下，前端界面必须围绕这些接口设计：

  1. 获取模型列表

  - GET /api/models
  - 用途：
    - 初始化模型选择器
    - 点击“刷新模型列表”时重新拉取

  2. 创建会话

  - POST /api/session
  - 请求体示例：
  {
    "model": "deepseek-chat"
  }
  - 用途：
    - 点击“新建会话”时创建会话
    - 使用当前所选模型创建会话

  3. 获取会话列表

  - GET /api/sessions
  - 用途：
    - 初始化左侧会话列表
    - 新建、删除、发送消息后刷新列表

  4. 删除会话

  - DELETE /api/session/{session_id}
  - 用途：
    - 点击会话项上的“删除”按钮
    - 删除前需要轻量确认提示 UI

  5. 获取会话历史

  - GET /api/session/{session_id}/history
  - 用途：
    - 点击某个会话时加载历史消息
    - 在主聊天区展示消息记录

  6. 流式发送消息

  - POST /api/message/async
  - 请求体示例：
  {
    "session_id": "xxx",
    "message": "你好"
  }
  - 返回类型：text/event-stream
  - 用途：
    - 前端点击“发送”后统一调用该接口
    - assistant 回复边接收边渲染
    - 流结束标记为 [DONE]

  7. 不需要接入的接口

  - POST /api/message
  - 该接口虽然存在，但本前端界面不需要使用，也不要为它设计交互模式

  ---
  五、关键交互要求

  请在生成页面时体现以下交互逻辑：

  1）新建会话

  - 点击新建会话
  - 使用当前已选模型创建一个新会话
  - 创建成功后自动选中该会话
  - 主聊天区切换到该会话

  2）切换会话

  - 点击左侧会话项
  - 加载对应历史消息
  - 在主区展示聊天记录

  3）删除会话

  - 点击删除
  - 先显示轻量确认
  - 删除成功后自动切换到相邻会话
  - 如果没有会话，则显示空状态

  4）发送消息

  - 用户输入消息后点击发送
  - 前端先立即在聊天区追加一条用户消息
  - 同时追加一条 assistant 占位消息
  - 然后通过流式接口不断增量更新 assistant 内容
  - 聊天区自动滚动到底部

  5）停止生成

  - 流式回复过程中，停止按钮可点击
  - 点击后中止当前流式展示
  - 即使先不实现真实取消协议，也要在界面层完整预留该状态与按钮行为

  6）模型选择

  - 顶部提供选择模型
  - 该选择只影响后续新建会话
  - 已创建会话应展示其所属模型，不因为顶部切换而被改写

  ---
  六、页面状态设计

  请把以下状态完整体现到 UI：

  1）初始加载状态

  - 模型选择器 skeleton
  - 会话列表 skeleton
  - 聊天区 skeleton

  2）空状态

  - 未选择会话时显示欢迎面板
  - 展示一句欢迎语和示例问题卡片

  3）无模型状态

  - 若模型列表为空，模型选择器禁用
  - 显示提示：当前暂无可用模型

  4）流式生成中状态

  - assistant 消息卡片显示“正在生成中”
  - “发送”按钮可变为 loading 或禁用
  - “停止”按钮高亮可用
  - 输入区域状态要清晰
  - 聊天区底部自动跟随最新内容

  5）错误状态

  请预留以下错误提示 UI：
  - 获取模型列表失败
  - 创建会话失败
  - 获取历史失败
  - 发送消息失败
  - 删除会话失败

  错误提示使用简洁的 toast 或 inline message，不要设计复杂弹窗。

  七、视觉风格要求

  请采用现代简约风格，参考主流大模型网页聊天界面，但不要直接照搬品牌视觉。

  色彩建议

  - 页面背景：浅灰或接近白色
  - 卡片背景：白色
  - 主强调色：克制、低饱和
  - 分割线/边框：极浅灰
  - 用户消息与助手消息做轻度视觉区分
  - 成功/错误/加载状态颜色要柔和

  字体与间距

  - 现代、简洁
  - 留白充分
  - 圆角适中
  - 阴影很轻
  - 页面克制，不要炫技，不要花哨渐变
